# R22 Migration and Rollback

## Scope

R22 is a code hardening release built on the R21 schema. It does not add `vm_top_current`, does not change the Agent payload, and does not introduce a new SQL migration. Existing migrations `001` through `015` remain canonical and are applied idempotently by the installer.

Supported paths:

- fresh install R22
- R20 to R22
- R21 to R22

## Pre-update checks

```bash
set -Eeuo pipefail

systemctl --no-pager --full status bw-monitor.service
systemctl list-units 'bw-monitor-maintenance@*.service' --state=running --no-legend
curl -fsS http://127.0.0.1:8080/healthz

df -h / /var/lib/docker /var/backups/bw-monitor
```

Do not update while a Maintenance worker is active.

## Automatic backups

`update.sh` now creates both:

1. a PostgreSQL backup through the installed `backup.sh`
2. `/var/backups/bw-monitor/pre-update-<UTC timestamp>/`

The pre-update directory contains:

- `application-source.tar.gz`, excluding only the virtual environment and generated bytecode
- application and PostgreSQL environment files when present
- credentials file when present
- systemd service unit
- Nginx site configuration when present
- `metadata.env` containing previous and target versions

Record the exact paths printed by the updater before continuing.

## Update

```bash
cd /path/to/virtinfra-monitor-r22
bash update.sh
```

The updater verifies the release checksum manifest, runs source preflight, creates backups, stops web/maintenance/retention services, applies the existing migrations, runs the Consumption backfill, installs code and performs health checks.

A failed backfill does not falsely become complete. Its status is stored under `consumption_backfill_status_r22` and appears in the existing database/Maintenance status card.

## Post-update verification

```bash
cat /opt/bw-monitor/DEPLOY_VERSION
curl -fsS http://127.0.0.1:8080/healthz
systemctl --no-pager --full status bw-monitor.service
journalctl -u bw-monitor.service -n 100 --no-pager
```

Verify through the UI:

- login and RBAC
- Dashboard
- Top VM sort by network, CPU, RAM, disk capacity and disk count
- Top VM Group/Node/search filters
- Consumption Node, Group, Summary and VM for 24H, 3D and 7D
- Storage I/O, Abuse VM and Node Health
- Maintenance Queue and backfill status

## Source-only rollback

Use this when migrations remain compatible and the problem is limited to R22 code.

```bash
set -Eeuo pipefail

SNAPSHOT=/var/backups/bw-monitor/pre-update-YYYYMMDDTHHMMSSZ
FAILED=/opt/bw-monitor.failed-r22.$(date -u +%Y%m%dT%H%M%SZ)

systemctl stop bw-monitor.service
mv /opt/bw-monitor "$FAILED"
install -d -m 0750 /opt/bw-monitor
tar -xzf "$SNAPSHOT/application-source.tar.gz" -C /opt/bw-monitor

# Keep the installed virtual environment if it was moved with the failed tree.
if [[ -d "$FAILED/venv" && ! -d /opt/bw-monitor/venv ]]; then
  mv "$FAILED/venv" /opt/bw-monitor/venv
fi

systemctl daemon-reload
systemctl start bw-monitor.service
curl -fsS http://127.0.0.1:8080/healthz
```

Restore configuration files from the snapshot only when the update changed them intentionally.

## Full database rollback

Use the installed restore command with the exact database backup created before update:

```bash
set -Eeuo pipefail

systemctl stop bw-monitor.service
virtinfra-monitorctl restore --from /var/backups/bw-monitor/<backup-directory> --with-config --yes
systemctl daemon-reload
systemctl start bw-monitor.service
curl -fsS http://127.0.0.1:8080/healthz
```

Database restore causes downtime and replaces post-backup data. Do not run it merely to undo a Python query change.

## Rollback verification

After either rollback, verify:

```bash
curl -fsS http://127.0.0.1:8080/healthz
systemctl is-active bw-monitor.service
journalctl -u bw-monitor.service -n 100 --no-pager
```

Then check Dashboard, Top VM, Consumption, agent push acceptance and Maintenance Queue.
